# 2026-05-21 — Workflow enhancement Tier 4b: frontend↔backend TS-type sync via openapi-typescript

## Headline
`src/types/api.ts` now generated from FastAPI's `app.openapi()` via the new `npm run gen:types` script. Closes the "HUD compiles fine but reads undefined" silent-break vector that §20 backend integrity discipline flagged but couldn't catch directly: contract drift on the wire (route paths + query params + response shapes when declared) now surfaces as TypeScript compile errors in the frontend instead of runtime undefined-access in production.

## Context

CLAUDE.md §20 "Frontend↔backend contract validation" anticipated this — `src/types/api.ts` generated from Pydantic models, with a pre-commit check that TS types match backend response shapes. Tier 4a landed `pre_commit_typecheck.py` (the second half of that closure); Tier 4b lands the type generation that the typecheck has something to verify against.

Trade-off accepted: FastAPI's openapi schema is only as good as the route declarations in `backend/api_server.py`. Most routes currently have no `response_model=` declared, so the generated types use `unknown`/loose objects for response bodies. The path inventory + query/path params + request bodies (where Pydantic-typed) are all strongly typed. Strengthening response types is incremental work — add `response_model=PydanticClass` on each route over time, re-run `npm run gen:types`, frontend reaps the type narrowing automatically.

## Resolution

### New dependency
- `openapi-typescript@^7.13.0` as devDependency (+ 19 transitive). Modifies `package.json` + `package-lock.json`. ~5MB to node_modules.

### New script
`scripts/dump_openapi.py` — invokes `from backend.api_server import app`, calls `app.openapi()`, writes JSON to `build/openapi.json` (gitignored — transient build artifact). Explicitly prepends repo root to `sys.path` so `python scripts/dump_openapi.py` invocation from npm works (Python's default sys.path[0] is the script's directory, not cwd — this was the first-run bug, fixed before commit).

### New npm scripts
```
"gen:openapi": "python scripts/dump_openapi.py"
"gen:types":   "npm run gen:openapi && openapi-typescript build/openapi.json --output src/types/api.ts"
```

### Generated baseline
`src/types/api.ts` (5118 lines, generated by openapi-typescript 7.13.0). Contains:
- `interface paths` — path inventory with method/parameter shapes for every `/api/*` route
- `interface operations` — operation IDs (with some duplicate-ID warnings flagged by FastAPI — see "Carry-forward" below)
- `interface components` — schemas for all Pydantic models exported through the OpenAPI surface

Committed verbatim as the initial baseline. Subsequent regenerations produce a diff that the operator reviews — that diff IS the contract change signal.

### Verified live
- `npm run gen:types` end-to-end → produces `src/types/api.ts` cleanly
- `npx tsc --noEmit -p tsconfig.json` → exits 0 (no type errors from the new types file)
- `python -m backend.diagnostics.capture_contracts diff` → CLEAN
- `python -m backend.diagnostics.pipeline_smoke verify` → CLEAN

### Build artifact handling
`build/openapi.json` written by `gen:openapi` is gitignored under the existing `build/` Python-dist pattern in `.gitignore` (line 43). Transient — regenerated on every `gen:types` invocation. Only `src/types/api.ts` is committed.

## Why it matters next time

Pre-Tier-4b, a backend dev could rename a Pydantic field on a `/api/cycles/last` response and the frontend would compile fine — only finding the breakage at runtime when `result.signals[0].confluence_breakdown` returned undefined. The `tsc --noEmit` pre-commit hook from Tier 4a now has something real to check: regenerate types after backend changes, and frontend code referencing removed/renamed fields fails to compile.

The wire-level contract is now triangulated three ways:
1. `capture_contracts diff` — schema snapshot (Python introspection)
2. `pipeline_smoke verify` — orchestrator config snapshot
3. `src/types/api.ts` regeneration diff — TS interface deltas

Different views of the same surface, all required to drift in lockstep when a backend contract intentionally changes.

## Carry-forward items

- **Duplicate operationId warnings.** FastAPI emits warnings on `get_cache_stats`, `get_cache_entries`, `clear_cache` — the same endpoints are defined twice (legacy `@app.get` in `api_server.py` + new `@router.get` in `routers/data.py`). Doesn't break gen:types, just produces a noisy stderr at every regeneration. The right fix is removing the legacy `@app.get` registrations now that the router is mounted. Out of Tier 4b scope; flagged for a separate "remove duplicate route registrations" cleanup pass.
- **Response-model strengthening.** Most routes return `JSONResponse` without a declared Pydantic response model. Generated TS uses loose types for those response bodies. Incremental work; not a blocker.
- **CI integration.** `npm run gen:types` is operator-driven today. A future pre-commit hook could regenerate + diff src/types/api.ts and refuse commits if backend contracts changed without regenerating types. Out of Tier 4b scope; flagged.
